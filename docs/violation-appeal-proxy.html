服务商违规申诉代办专区：代申诉能力开放方案

服务商不能绕过商家直接提交；服务商负责专业处理，商家保留最终确认权。

1 背景与目标

服务商现在已经在协助商家处理小程序违规，但很多动作还在线下完成：商家发现违规后找服务商，服务商再整理材料、写申诉说明、指导或登录 MP 后台处理。这个链路依赖人工沟通，处理状态不集中，关键操作也缺少统一记录。

本项目在服务商平台建设违规申诉代办专区，让服务商集中查看违规、跟进整改和处罚状态，并在商家确认后提交申诉。

当前现状问题影响
商家收到违规后再找服务商服务商被动响应容易错过警告整改和申诉窗口
服务商指导商家去 MP 后台处理协同成本高处理结果不同步，服务商难持续跟进
商家自行申诉申诉质量不稳定可能浪费申诉次数
场景本版目标产品口径
警告整改降低警告升级为处罚的风险警告不进入申诉链路，只引导客服在整改期限内完成整改
处罚申诉提升处罚后的处理和跟进效率处罚类先整改，再按平台规则提交 / 跟进申诉
队列收敛降低客服在多入口间跳转的成本用 主列表 + 聚合弹窗 + 链接 承载主流程

问题规模与不做的代价（数据来源：2025-10 ~ 2026-03 违规数据）：

指标 数据 含义
服务商功能封禁月均量 2.0 ~ 3.5 万起（占全量 20-29%） 服务商违规主战场在功能类处罚，不是账号封禁
功能警告 → 封禁升级率 服务商 47-58%，高于非服务商 4-9 个百分点 服务商警告期整改力度不足，升级风险更高
功能封禁平均恢复时长 服务商 7.50 天 vs 非服务商 7.11 天（2026-03） 服务商比大盘多花 0.4 天，有下降空间
功能封禁恢复率 服务商 15-30%，高于非服务商 6.5-16%，但恢复时长更长 服务商申诉能力本身不弱，瓶颈在流程效率而非申诉质量
上线目标：恢复时长保底对齐非服务商（6.65 天），推荐 5.8-6.2 天；升级率保底对齐非服务商（44.8%），推荐 40-42%。
边界声明：服务商不能绕过商家直接提交。代申诉的本质是服务商负责专业处理，商家保留确认权。

3 调研情况

3.1 问卷情况

问卷说明，服务商已经在处理大量违规相关事项，且处理所需信息多数来自服务商已有资料或商家本次提供。真正需要产品补齐的，不是让服务商"第一次接触这些信息"，而是把商家确认、次数提示、操作记录和双方通知做成标准流程。

问卷发现对项目的含义
80.3% 会接触违规记录、原因、整改指引、申诉状态等处理信息；73.8% 会接触商家主体/经营材料代申诉需要的信息本来就在服务商日常处理范围内
违规处理信息主要由商家在本次处理中提供（75.4%），主体/经营材料也主要由商家在本次处理中提供（67.2%）；一部分来自服务商原本可见或已持有多数材料可由"服务商已有 + 商家本次提供 + 处理过程生成"覆盖，不需要平台额外拉取大量新信息
仅 15% 不到认为不需要通知商家绝大部分服务商本身认为商家确认是必要环节
0% 选择"商家不可以停止授权、撤回处理或接管申诉"服务商普遍认可商家保留控制权
申诉异常后，77.0% 会继续协助整改/补件/重新申诉，75.4% 会通知商家状态，63.9% 会要求商家补充或确认材料申诉不是提交后就结束，服务商仍会参与后续处理

3.2 访谈情况

访谈说明，不同服务商处理能力不一样，但共同点是：服务商希望更早看到违规、更快判断处理路径，并减少商家来回沟通成本。

访谈收获对项目的含义
有赞、爨火都会介入旗下小程序违规处理服务商有动力也有能力处理一部分违规问题
当前很多处理仍是商家来找服务商后才开始监管专区应帮助服务商主动发现违规，而不是只等商家求助
多数常见申诉材料服务商可以拿到，或可由商家在本次处理中补充代申诉不要求平台额外收集大量新材料，重点是把材料整理和商家确认放到线上
警告期前置整改优先于处罚后申诉项目不能只做申诉，也要支持查看指引、整改跟进
代申诉消耗商家次数不是绝对阻碍关键是提交前让商家明确知道并确认
不宜完全冻结商家自行申诉商家和服务商都可处理，但同一条违规不能重复提交
服务商差异
有赞多数常见整改和申诉材料组织可由服务商完成；常见材料如小程序截图通常可获取，主体材料等复杂情况需要商家配合或内部确认
爨火已有常见违规处理流程和模板；多数客户从注册开始由爨火接手，相关资料和文件服务商侧已有掌握
银豹更偏向给商家指引，不明显承担完整代处理流程

综合问卷和访谈，服务商已经具备参与违规处理的基础，但不同服务商能力不同。产品设计要解决的关键问题不是"服务商会不会参与"，而是"服务商提交前，商家是否明确知情并确认"。这一结论也是给法务的核心依据：在商家知情确认、授权范围清楚、系统有记录、页面有责任提示的前提下，代申诉风险可控。

调研发现 → 设计决策映射：

调研发现设计决策
80.3% 已接触违规信息，信息本身不是瓶颈专区不做信息拉取，做信息聚合和流程线上化
0% 认为商家不能撤回/取消授权商家 H5 保留拒绝权，确认后才提交、才消耗次数
73.8% 已接触材料，但需商家本次补充表单只要求申诉说明和证明材料，不额外拉取商家基础信息
警告期前置整改是更核心的需求专区同时覆盖警告型和处罚型两条路径，不只做申诉

4 流程与状态

4.1 处理路径总览

违规记录按类型分为警告型和处罚型，决定是否进入申诉链路：

类型触发记录是否可申诉服务商动作
警告型警告提醒否查看原因、跟进整改、关注截止时间
处罚型功能限制、账号限制、页面封禁、下架是查看处罚原因、完成整改、准备材料、提交/跟进申诉
边界：警告升级为处罚后，原警告记录进入已处理，处罚记录作为新记录独立计数和展示。两条记录之间不做自动关联。

4.2 代申诉确认流程

处罚型记录中，服务商代填申诉说明和材料后，需经商家 H5 确认才能正式提交。泳道图展示这一核心交互：

蓝色为主流程（服务商代填 → 系统冻结自申诉并发 H5 → 商家查看）；绿色为商家确认提交；橙色为商家拒绝；红色为 24h 超时未处理。次数判定见下方规则，全程双向同步。

核心规则要点：

规则说明
服务商只能代填，不能直接提交填写后只生成待商家确认任务，不进入审核队列
商家确认后才提交H5 查看内容和次数提示后点确认，才进入审核队列并扣减申诉次数
确认前不消耗次数发起确认、未确认、拒绝或超时均不消耗
H5 确认 24h 有效超时自动失效，超时时间自服务商发起时起算 24 小时
同一条违规不能重复提交已有审核中申诉时另一方不能提交
待确认期间冻结商家自申诉后端校验拦截（非前端 UI 冻结）；不冻结商家查看违规或处理其他违规
超时后恢复提交入口双方提交入口恢复，状态变更且清除冻结
全程双向同步违规与审核结果走原有微信通知与 API 回调；代申诉确认链路新增 H5 与回调事件；各阶段触达分工见第 6 节

边界：

边界设计
服务商能否撤销已发起的确认不支持。发起后只能等商家确认、拒绝或 24h 超时
拒绝后重新发起的次数限制不限制。服务商可反复整理后重新发起，商家可反复拒绝
商家自行提交的申诉内容服务商可查看（已有能力），但不能代为操作

4.3 状态体系

状态按「记录类型 → 主状态 → 子状态」三层组织，共 2 类、7 个主状态、3 个子状态。x/2 表示剩余申诉次数 / 总申诉次数。

4.3.3 代申诉确认子状态

「待首次提交申诉」和「驳回后可再提交」在服务商发起确认后进入子状态；子状态不改变处理状态归属（仍在待处理），只改变列表行文案。

子状态触发状态文案操作入口期间行为结束后

4.4 处理状态归属规则

列表不分页签，用「处理状态」筛选项组织记录。待处理、申诉中、已处理是记录的归属分类，可与搜索、限制类型、时间筛选在同一列表内联合生效。

处理状态定义包含记录数量展示
待处理当前仍未恢复或仍需服务商关注警告待整改、待首次提交申诉、驳回后可再提交、申诉次数耗尽展示数量角标
申诉中已提交申诉，等待平台审核结果申诉审核中的处罚记录展示数量角标
已处理当前无需继续处理申诉成功已恢复、警告整改期结束不展示数量

进入页面和处理状态筛重置后，默认选中「待处理」。筛选另提供「全部」，便于跨状态检索。数量角标具体展示位置以设计稿为准。

警告类记录在整改截止前属于待处理；到整改截止时间后自动进入已处理。若后续处罚落地，处罚作为新的违规记录进入队列。申诉次数耗尽 虽然没有可再提交的动作，但限制尚未恢复，仍留在待处理，入口只允许查看原因、最近一次申诉和指引，不再展示提交按钮。

6 通知流程

商家侧在原有微信通知链路基础上，仅代申诉确认环节新增 H5；服务商侧通过微信 API 回调、模板消息和监管专区控制台三轨触达。不自建通知推送系统。

侧通知对象
小程序商家管理员、运营者、其他单独开通违规处理权限的人员
服务商V1：运营者；权限模块上线后：具备违规申诉待办专区写入权限的人员

通知对象与可操作身份一致：仅向具备相应权限的人员触达，H5 OAuth 亦仅允许同一范围身份操作。
