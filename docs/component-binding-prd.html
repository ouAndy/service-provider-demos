第三方平台账号绑定 — 方案

日期：2026/07/12 · 读者：服务商平台产品、设计、研发、测试 · 状态：已定稿（2026/07/12：去掉所属 Open、绑定时间；占用方名称不展示、接口不返回；小程序数页内 enrichment；文案「在服务商平台使用能力」）

1. 背景、问题与范围

已定稿：2026/07/13：绑定后消费按模块分化（代申诉合并、审核管理切换账号）；写操作落到小程序所属第三方平台。占用方名称不展示；小程序数页内 enrichment；按需鉴权 authorizer-auth 已落地。

做什么 / 为什么

服务商与小程序的关系沉淀在开放平台：第三方平台账号持有商家授权的小程序。服务商平台要做代申诉、审核加急等能力，须先明确：当前服务商平台账号认领了哪些第三方平台账号，再按该映射去开放平台拉取小程序关系。

第三方平台账号与服务商平台账号同属一个服务商主体。绑定是主体内部的账号认领，不创造新的小程序授权；开放平台仍是小程序关系的唯一权威。服务商仍可通过开放平台 API 自行使用相关能力；本页绑定只为在服务商平台内使用这些能力。

绑定与解绑在服务商平台完成；开放平台本期不提供绑定操作界面。可复用「拉取同主体第三方平台账号」能力；不复用流量主代运营的模块内绑定界面与主体共享模型。

成功标准

管理员能在服务商平台完成认领与释放；业务模块只消费已认领范围内的小程序数据；同主体下同一第三方平台账号不会被两个服务商平台账号同时认领。

本期做 / 不做

  • 做：关联页直接勾选绑定/解绑；分页；入驻或首次登录引导；业务模块未认领时空态引导；统一池消费
  • 不做：单独绑定弹窗；列表搜索；与流量主绑定数据打通；开放平台绑定操作界面；点行查看第三方平台详情；因在途单据系统拦截解绑；强制把认领关系同步到开放平台（是否同步由研发评估，见附录 C）

同步开放平台—产品不强制本期同步；是否同步由研发评估（附录 C）

*ps.* 占用互斥的意思是：同一个第三方平台账号不能同时被两个服务商平台账号绑定；一个服务商平台账号可以绑定多个第三方平台账号。绑/解绑仅企业管理员可操作，是因为现网只能按管理员等角色控权，还做不到按菜单单独授权。

3. 页面与交互

路径：平台管理 → 第三方平台账号关联

本页一层完成查看与绑定：不做单独绑定弹窗。

3.1 全页共用

  • 权限：管理员可勾选、确认绑定、解绑；非管理员只读；关联菜单对已加入企业的成员均可见
  • 页头说明：绑定后，可在服务商平台使用这些账号下小程序的相关能力（如代申诉、审核加急）。不影响通过 API 使用，也不影响流量主代运营及小游戏、小店相关业务。仅面向小程序服务商。
    • 口径：绑定只为在服务商平台内使用相关能力；服务商仍可通过开放平台 API 自行调用，不依赖本页绑定
    • 当前账号尚未绑定任何第三方平台时，可追加一句：未绑定则代申诉、审核加急等能力在服务商平台无数据且无法使用
    • 「仅面向小程序服务商」为说明文案，不按服务商类型隐藏菜单或引导
    • 目标态定义（本期可不落标签）：在本页完成第三方平台账号绑定的服务商，视为小程序服务商
  • 筛选：全部（默认）｜可绑定｜已绑定｜被占用；只过滤；切换筛选时清空勾选；筛选与分页在服务端完成
  • 排序（全页统一，与筛选无关）：
    • 切页前按 AppID / 名称；仅对当前页补关联小程序数后，页内再按数量从多到少（相同则 AppID、名称）
    • 数量失败（--）排在有数之后
    • 账号很多时跨页不保证全局按数量序（避免全量 RPC）；多数企业 ≤10 个 T 时页内序 ≈ 全局序
    • 前端按返回顺序展示，避免异步加载导致列表跳动
  • 不做：单独绑定弹窗；列表搜索；列表不展示「所属 Open」「绑定时间」「占用方名称」（认领决策不依赖；绑/解绑时间只落库审计）
  • 列表加载失败：沿用服务商平台现有列表失败态（含重试）；不与空态「暂无数据」混用

3.2 账号列表

同主体全部第三方平台，一行一个：

勾选仅「可绑定」可勾选（管理员）；已绑定、被占用置灰不可勾选；非管理员无勾选

AppID第三方平台 AppID

状态可绑定 / 已绑定 + [解绑]（仅管理员，同行展示）/ 被占用（不展示占用方名称）

同主体下没有任何第三方平台

  • 空态文案：WeUI 默认「暂无数据」（不单独定制；与平台其它列表一致）
  • 无确认绑定按钮

有账号但当前筛选无匹配

  • 空态文案：同上，WeUI「暂无数据」
  • 确认绑定按钮隐藏或禁用

从其他入口进入

  • 入驻引导、代申诉等空态的「去绑定」：跳转到本关联页（本页即勾选绑定，不再另开弹窗）

3.3 绑定二次确认

  • 触发：管理员已勾选 ≥1 项，点 [确认绑定]
  • 标题：确认绑定
  • 正文：将绑定已选的 N 个账号。绑定后可在服务商平台使用其下小程序的相关能力（如代申诉、审核加急）。
  • 按钮：[确定]（可关弹窗；无单独取消按钮亦可）
  • 全部成功：Toast「绑定成功」→ 刷新列表（对应行变为已绑定，勾选清空）
  • 并发冲突（部分被抢）：已抢到的写入成功；Toast「部分账号已被占用，未能完成绑定，请重试。」→ 整表刷新；前端不依赖接口返回逐条成功/失败名单

3.4 解绑确认

  • 标题：确认解绑
  • 正文：解绑后，将无法在服务商平台查看该账号下的小程序及相关数据。已提交的申诉、审核等单据不受影响。请确认当前没有需要继续跟进的事项。
  • 按钮：[确定]（危险确认；可关弹窗）
  • 成功：Toast「已解绑」→ 关闭确认框 → 刷新列表（该行变为可绑定）
  • 已打开的代申诉等业务页：须在再次进入或切回时重拉统一池；禁止继续按解绑前的旧列表提交新操作

3.5 入驻 / 首次登录引导

  • 触发：入驻完成或首次进入，且同主体下存在至少一个第三方平台账号，且当前账号尚未绑定
  • 频次：最多出一次；点「稍后再说」或关闭后不再弹出
  • 不出：同主体下没有任何第三方平台时
  • 标题：小程序服务商绑定第三方平台账号
  • 正文：面向小程序服务商。绑定后可在服务商平台使用代申诉、审核加急等依赖小程序授权关系的能力。可稍后在「第三方平台账号关联」中完成。
  • 管理员：[去绑定]（进入关联页）+ [稍后再说]
  • 非管理员：仍出引导（满足触发条件时）；无「去绑定」；正文提示联系管理员；可「稍后再说」

3.6 业务模块未绑定态

依赖小程序授权关系的能力（如代申诉、审核加急等）：

  • 管理员：空态 + 引导至关联页
  • 非管理员：空态 + 「请联系管理员绑定」
  • 不做第三方平台切换器；列表不展示数据来自哪个第三方平台
  • 举例：代申诉能力最先按上述空态展示；审核加急等同批能力一并接入

*ps.* 页头与引导文案与上述能力名称对齐；其他模块后续接入时同样走统一池与未绑空态。

A. 本期边界

  • 本期上线
    • 关联页：同主体列表直接勾选绑定、状态筛选、按小程序数排序、解绑确认、每页 10 条分页（含跳转）
    • 入驻/首次引导（最多一次；无第三方平台则不出）
    • 业务模块未认领空态引导
    • 已绑池供业务模块按自身规则消费；写操作落到小程序所属第三方平台
    • 占用互斥与冲突提示；绑/解绑审计（只落库，无前台审计页）
  • 二期
    • 点行查看第三方平台详情；批量解绑；在途单拦截解绑；前台审计查询；列表搜索（见附录 B）
  • 不做
    • 单独绑定弹窗；本期列表搜索
    • 关联页列表展示「所属 Open」「绑定时间」（二期详情或审计查询如需再议）
    • 与流量主绑定数据打通；开放平台绑定操作界面；生效开关；按 Open 账号分 Tab
    • 未认领时拉取同主体全部第三方平台（业务模块侧）
    • 列表加载失败的专用文案（沿用平台现有列表失败态即可）
  • 复用
    • 复用：拉取同主体第三方平台账号的能力
    • 不复用：流量主 iframe 内 UI、主体账号池共享绑定模型
  • 连带影响
    • 各业务模块按自身规则消费已绑池（代申诉合并展示且可不做切换器；审核管理等可保留当前第三方账号切换）；不强制所有模块同一套读侧 UI
    • 与流量主代运营、小游戏、小店：不打通、不迁移、不互斥校验；本页映射只服务代申诉、审核加急等平台级能力。页头说明不影响通过 API 使用，也不影响流量主代运营及小游戏、小店业务
  • 上线节奏
    • 关联页可先于代申诉上线，便于提前触达与习惯养成；代申诉等消费模块随后接入统一池即可，不要求同版本强绑定，但不宜长期双轨取数
  • 相关另案
    • 准入门槛（认证 Open → 入驻 → 保证金 → 开通第三方平台能力）不在本期落地
    • 第三方平台因处罚限增等对存量能力的影响本期不纳入（限增管增量，存量授权关系仍在；商家解授权才减少可见小程序，服务商不能单方面解除授权）
    • 「小程序服务商」身份标签若产品化，以本页是否已绑定第三方平台账号为准；本期不做类型分流入口
    • 不按「第三方平台账号被删除/失效」设计主流程：现网无产品化删 T；Open 账号注销为低频人工路径且须先清名下应用。本期不做「已失效」行态与自动清映射；列表与消费以 Open 拉到的现存关系为准

B. 二期草案

  • 已绑行点开详情：展示该第三方平台下小程序摘要等
  • 批量解绑
  • 解绑前校验在途单据并系统拦截（若业务需要）
  • 前台审计查询页（绑/解绑操作记录）
  • 列表按名称 / AppID 搜索（若尾部主体反馈强烈再补）
